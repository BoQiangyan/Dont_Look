# 《冷不丁：背后梆梆拳》开发文档

## 1. 项目定位

《冷不丁：背后梆梆拳》是一款真人玩家对战系统 AI 的客户端小游戏。第一版优先面向手机端 H5 / App WebView / 小程序容器，而不是电脑键盘操作。画面采用竖屏上下对峙 UI：AI 对手在上方，真人玩家在下方，每个角色背后各有一台投掷机。玩法不是实时动作闯关，而是同步回合制心理博弈。

游戏核心体验是：玩家一边防正面的拳，一边防背后的投掷机。每回合双方同时选择行动，结算后播放夸张动画和字幕，制造“冷不丁被梆梆”的喜剧效果。

## 2. 核心卖点

- 第一版单人可玩，真人玩家对战系统角色。
- 用界面按钮选择行动，适合触屏客户端。
- 后续可升级为本地双人或联网双人对战。
- 操作简单，每回合只需选择一个行动。
- 规则像石头剪刀布，但多一层“正面/背后”的判断。
- 竖屏街机对峙表现，强化“正面拳 + 背后投掷机”的荒诞反差。
- 每局时长短，适合快速重开。

## 3. 推荐玩法结构

### 3.1 对战人数

- 第一版：1 名真人玩家 + 1 名系统 AI。
- 第二版：联网双人对战。
- 本地同屏双人不作为第一目标，因为手机端不适合两个人同时抢同一块屏幕操作。

### 3.2 对战视角

- 竖屏 2D。
- AI 对手位于上方，真人玩家位于下方。
- 对手背后是屏幕更上方，玩家背后是屏幕更下方。
- 每个角色背后各有一台投掷机。
- 玩家需要同时判断对方的正面动作和自己背后的威胁。

### 3.3 回合流程

每回合分为 4 个阶段：

1. 准备阶段  
   显示双方当前血量、投掷机冷却、冷不丁值。

2. 选择阶段  
真人玩家在倒计时内点击界面按钮选择行动，系统 AI 同时根据当前局势选择行动。玩家不能提前知道 AI 的选择。

3. 揭示阶段  
同时展示玩家和 AI 的行动。

4. 结算阶段  
根据行动组合判定命中、闪避、受伤、冷却和资源变化，并播放动画。

## 4. 行动设计

### 4.1 当前 MVP 行动

当前 MVP 使用 4 个基础行动：

| 行动 | 作用 | 克制 | 被克制 | 备注 |
| --- | --- | --- | --- | --- |
| 等待 | 准备闪避正面拳 | 打拳 | 投掷机 | 也是恢复节奏的动作 |
| 看后面 | 防御背后投掷机 | 投掷机 | 打拳 | 看后面时正面露破绽 |
| 打一拳 | 正面攻击 | 看后面、操控投掷机 | 等待 | 基础伤害低，但节奏快 |
| 操控投掷机 | 背后攻击 | 等待、打拳 | 看后面 | 伤害高，但有冷却 |

### 4.2 v0.2 待审行动方案

试玩反馈：`等待`的用途偏窄。它只防正面拳，但面对投掷机时仍然吃亏，导致玩家会觉得它不如直接打拳或看后面。

v0.2 建议删除`等待`按钮，把游戏核心压缩成 3 个常用行动 + 1 个特殊行动：

| 行动 | 作用 | 主要风险 | 备注 |
| --- | --- | --- | --- |
| 看后面 | 防投掷机，成功时获得 1 点冷不丁值 | 会被正面拳打中 | 这是“防背后”的核心动作 |
| 打一拳 | 正面攻击，抓对方看后面或操控机器 | 遇到投掷机会互伤 | 保留主动进攻感 |
| 操控投掷机 | 背后攻击，打没看后面的人 | 有冷却，被看后面完全躲掉 | 伤害高但不能连发 |
| 冷不丁两拳 | 冷不丁值满后可用的强正面攻击 | 被“看后面”以外的防御机制暂不处理 | 后续可再决定是否需要专门反制 |

v0.2 结算重点：

- `看后面`遇到`投掷机`：看后面方闪避成功，并获得冷不丁值。
- `看后面`遇到`打一拳`：看后面方挨拳。
- `打一拳`遇到`操控投掷机`：双方互伤；拳造成 1 点，投掷机造成 2 点。
- `打一拳`遇到`打一拳`：双方对拳，各受 1 点，或者改成双方弹开不掉血，待测试决定。
- `投掷机`遇到`投掷机`：双方都被背后砸中，各受 2 点。

这个方案让玩家每回合只思考一个问题：`我现在是看背后、打正面，还是赌一把机器？`

## 5. 克制关系

核心闭环：

```text
等待 克 打拳
打拳 克 看后面
看后面 克 投掷机
投掷机 克 等待
```

这个闭环让每个行动都有价值，也都有风险。

## 6. 结算规则

### 6.1 基础伤害

- 打一拳：造成 1 点伤害。
- 投掷机：造成 2 点伤害。
- 冷不丁两拳：造成 2 点伤害。

### 6.2 基础血量

- 每名玩家初始 5 点血。
- 血量归零的一方失败。
- 同回合双方血量同时归零时，判定为平局或进入“最后一拳”加赛。

### 6.3 行动结算表

| A 行动 | B 行动 | 对 B 的结果 | 对 A 的结果 |
| --- | --- | --- | --- |
| 打一拳 | 等待 | B 闪避成功 | A 打空，进入硬直 |
| 打一拳 | 看后面 | B 受到 1 伤害 | A 命中 |
| 打一拳 | 操控投掷机 | B 受到 1 伤害 | 投掷机是否命中按 B 的投掷结算 |
| 打一拳 | 打一拳 | 双方各受 1 伤害 | 双方各受 1 伤害 |
| 投掷机 | 等待 | B 受到 2 伤害 | A 投掷机进入冷却 |
| 投掷机 | 看后面 | B 闪避成功 | A 投掷机进入冷却 |
| 投掷机 | 打一拳 | B 受到 2 伤害 | A 受到 1 伤害 |
| 投掷机 | 投掷机 | 双方各受 2 伤害 | 双方各受 2 伤害 |
| 等待 | 等待 | 双方获得 1 点冷不丁值 | 双方获得 1 点冷不丁值 |
| 看后面 | 看后面 | 无事发生 | 无事发生 |

说明：表格是从 A 对 B 的角度写的，实际结算时双方行动同时生效。

## 7. 平衡参数

第一版参数建议：

| 参数 | 初始值 |
| --- | --- |
| 初始血量 | 5 |
| 拳头伤害 | 1 |
| 投掷机伤害 | 2 |
| 投掷机冷却 | 1 回合 |
| 冷不丁值上限 | 3 |
| 冷不丁两拳伤害 | 2 |
| 每回合选择时间 | 5 秒 |
| 打拳被闪避后的硬直 | 1 回合不可打拳 |

## 8. 冷不丁值系统

冷不丁值用于防止游戏变成单纯猜拳，并鼓励玩家制造节奏变化。

### 8.1 获得方式

- 双方都等待：双方各 +1。
- 成功闪避对方打拳：闪避方 +1。
- 成功看后面躲开投掷机：躲避方 +1。
- 打拳被闪避：攻击方不获得。

### 8.2 消耗方式

当冷不丁值达到 3，玩家可以选择特殊行动：

| 特殊行动 | 效果 | 被克制 |
| --- | --- | --- |
| 冷不丁两拳 | 正面造成 2 点伤害 | 等待 |

特殊行动命中后清空冷不丁值。被等待闪避后也清空，制造风险。

## 9. MVP 范围

第一版只做最小可玩闭环：

- 真人玩家对战系统 AI。
- 手机端触屏操作。
- 竖屏街机对峙场景。
- 两个角色和两台投掷机。
- 每回合同时选择行动。
- 4 个基础行动。
- 血量、投掷机冷却、硬直、冷不丁值。
- 行动揭示和结算字幕。
- 胜负判定。
- 一键重开。

第一版暂不做：

- 联网对战。
- 复杂地图。
- 多角色技能。
- 道具系统。
- 排行榜。

## 10. 客户端交互方案

### 10.1 真人玩家行动按钮

玩家通过屏幕底部的行动按钮选择本回合动作。

| 按钮 | 行动 | 状态 |
| --- | --- |
| 看后面 | 防御背后投掷机 | 始终可用 |
| 等待 | 准备闪避正面拳 | 始终可用 |
| 打一拳 | 正面攻击 | 硬直时禁用 |
| 操控投掷机 | 背后攻击 | 冷却时禁用 |
| 冷不丁两拳 | 特殊攻击 | 冷不丁值满时可用 |

### 10.2 选择交互

- 每回合底部出现 4 到 5 个行动按钮。
- 玩家点击后按钮高亮，并显示“已选择”状态。
- 在倒计时结束前，玩家可以改选。
- 倒计时结束后锁定选择并进入揭示阶段。
- 如果玩家未选择，默认执行“等待”。
- 被硬直或冷却限制的按钮置灰，并显示剩余回合数。

### 10.3 系统 AI

系统 AI 不需要键盘输入。每回合在玩家选择阶段同时决策。

第一版 AI 不追求复杂智能，只需要让玩家感觉“它会猜、会变招、偶尔很欠”。

## 11. AI 设计

### 11.1 AI 目标

AI 的目标不是完美获胜，而是制造喜剧节奏和可读的心理压力。它需要做到：

- 不总是随机。
- 不总是最优。
- 会根据玩家上一回合行动调整策略。
- 偶尔连续使用同一种套路，让玩家产生“我看穿你了”的感觉。
- 偶尔突然变招，制造冷不丁效果。

### 11.2 AI 难度

第一版建议做 3 个难度：

| 难度 | 行为特点 |
| --- | --- |
| 简单 | 主要随机，偶尔克制玩家上一回合 |
| 普通 | 会统计玩家最近 3 回合偏好，并提高对应克制行动概率 |
| 欠揍 | 更爱投掷机和冷不丁两拳，但仍保留失误概率 |

### 11.3 AI 决策逻辑

普通难度可以用权重系统实现：

1. 给所有合法行动一个基础权重。
2. 根据玩家上一回合行动，提高克制行动的权重。
3. 根据 AI 自己的投掷机冷却、硬直、冷不丁值过滤非法行动。
4. 根据血量差调整进攻或防守倾向。
5. 按权重随机选择。

示例：

| 玩家上一回合 | AI 倾向 |
| --- | --- |
| 等待 | 更可能操控投掷机 |
| 看后面 | 更可能打一拳 |
| 打一拳 | 更可能等待 |
| 操控投掷机 | 更可能看后面 |

### 11.4 AI 性格包装

AI 不要表现得像“电脑”。它可以被包装成一个会嘴硬的对手。

结算后可随机出现短句：

- “你回头啊。”
- “我没动，是机器动的。”
- “刚刚那拳是风。”
- “这叫战术。”
- “你别老看后面。”

## 12. 画面设计

### 12.1 主界面

第一屏直接进入游戏，不做营销式首页。

画面结构：

- 顶部：AI 血量、冷不丁值、投掷机冷却。
- 中上：AI 投掷机和 AI 角色。
- 中间：回合字幕、命中特效、投掷物轨迹。
- 中下：玩家角色和玩家投掷机。
- 底部上沿：玩家血量、冷不丁值、投掷机冷却。
- 底部：当前回合倒计时、行动按钮、玩家选择状态。

### 12.2 手机端布局

推荐竖屏优先。第一版不强求横屏适配，只保证横屏时界面不崩坏。

竖屏结构：

```text
┌────────────────────┐
│ AI 血量 / 冷不丁值 / 冷却 │
├────────────────────┤
│       AI 投掷机      │
│        AI 角色       │
│                    │
│   梆！/闪避/被砸中    │
│                    │
│       玩家角色       │
│      玩家投掷机       │
├────────────────────┤
│ 玩家血量 / 冷不丁值 / 冷却 │
│       倒计时         │
├────────────────────┤
│ 看后面  等待         │
│ 打一拳  投掷机       │
│ 冷不丁两拳           │
└────────────────────┘
```

横屏结构：

- 上方显示状态栏。
- 中间保留上下对峙关系，不改成左右对站。
- 右下角放行动按钮组。
- 左下角放倒计时和回合提示。

第一版建议先做竖屏，因为更符合客户端小游戏的自然使用习惯。

### 12.3 竖屏“背后”表达

- 玩家角色的正面朝上，背后朝下；AI 角色的正面朝下，背后朝上。
- 玩家选择“看后面”时，角色转身向下看自己的投掷机。
- AI 选择“看后面”时，角色转身向上看自己的投掷机。
- 玩家“打一拳”时向上突进，AI“打一拳”时向下突进。
- 玩家操控投掷机时，AI 背后的机器启动，从 AI 后方砸向 AI。
- AI 操控投掷机时，玩家背后的机器启动，从玩家后方砸向玩家。

### 12.4 美术方向

- 复古街机像素风。
- 角色动作夸张，拳头命中时要有明显顿帧。
- 投掷机可以投出桶、砖块、沙袋等荒诞物件。
- 结算字幕要短促、有节奏。

示例字幕：

- “看后面！”
- “没看！”
- “梆！”
- “梆梆！”
- “打空了！”
- “被背后砸中！”
- “有破绽！”

## 13. 技术方案

### 13.1 推荐实现

第一版推荐做成移动端 H5 小游戏，方便后续嵌入客户端、App WebView 或小程序 WebView：

- Vite
- TypeScript
- Canvas 或 Phaser

如果追求快速出原型，优先 Phaser，因为它自带场景、动画、输入和资源管理。

如果目标客户端是微信小游戏或字节小游戏，后续可以再评估适配小游戏运行时；第一版先用标准 H5 跑通规则和体验。

### 13.2 核心模块

```text
src/
  main.ts
  game/
    constants.ts
    ai.ts
    types.ts
    rules.ts
    state.ts
    input.ts
    touchControls.ts
    scene.ts
    ui.ts
```

### 13.3 规则层设计

规则判断应独立于画面，方便后续平衡。

关键函数：

```ts
resolveRound(playerAAction, playerBAction, currentState): RoundResult
```

输入双方行动和当前状态，输出：

- 双方伤害
- 是否闪避
- 是否硬直
- 投掷机冷却变化
- 冷不丁值变化
- 本回合字幕
- 需要播放的动画

## 16. 联网双人模式预留设计

为了第二版不重写核心代码，第一版就要把“谁在控制角色”抽象出来。

推荐接口：

```ts
type ControllerType = "human" | "ai" | "remote";

interface Controller {
  type: ControllerType;
  chooseAction(state: GameState, playerId: PlayerId): Promise<Action>;
}
```

第一版实现：

- `HumanController`
- `AIController`

第二版增加：

- `RemoteController`

这样规则层只关心双方最终选择了什么行动，不关心行动来自界面按钮、AI 还是网络。

## 17. 风险和待验证问题

- 投掷机 2 点伤害可能偏强，需要冷却限制。
- “等待”如果收益太高，玩家可能过度保守，需要通过投掷机克制。
- “看后面”如果只防投掷机，使用频率可能不足，可以让成功看穿投掷机获得冷不丁值。
- 特殊行动可能导致滚雪球，需要命中和失败都清空冷不丁值。
- 同时结算时，互伤场景要足够清晰，否则玩家会觉得判定不透明。
- AI 如果太随机，游戏会没有心理博弈感。
- AI 如果太会克制，玩家会觉得被系统作弊。
- 联网双人有额外工程成本，应等单机玩法验证后再做。

## 18. 下一步建议

下一步可以直接进入 Version 0.1 开发。建议先用 Phaser 做移动端 H5 版本，优先完成规则层、AIController、触屏行动按钮和一个无素材的可玩版本，再替换成像素风角色和投掷机素材。
